Планировщик Go (GMP-модель) управляет тысячами легковесных горутин (G) на ограниченном числе системных тредов (M), используя процессоры (P) для эффективного распределения задач, чтобы обеспечить конкурентность и параллелизм. Он эволюционировал от наивной модели 1:1 к сложной M:N с оптимизациями для минимизации переключений контекста и простоя ядер CPU.

## Основные компоненты

- **Горутины (G)**: Легковесные задачи в состояниях Runnable (в очередях), Executing (на треде) или Waiting (блокировка, напр. syscall).
    
- **Треды (M)**: Системные потоки ОС, ограничены GOMAXPROCS (по умолчанию = NumCPU()), выполняют горутины; пул для переиспользования.
    
- **Процессоры (P)**: По одному на тред, управляют очередями горутин; количество = GOMAXPROCS.
    

## Очереди и распределение

- **LRQ (Local Run Queue)**: Локальная очередь каждого P — быстрая, без мьютекса.
    
- **GRQ (Global Run Queue)**: Общая очередь с мьютексом для новых/непривязанных горутин; проверяется редко (1/61 итераций).
    
- Алгоритм P: LRQ → steal у других P (work stealing, половина очереди) → GRQ → netpoller.
    

## Ключевые механизмы

- **Work stealing**: P крадёт горутины из LRQ других для баланса нагрузки.
    
- **Handoff**: При блокирующем syscall тред+G отвязываются, создаётся новый M; Sysmon мониторит таймауты (10 мс).
    
- **Netpoller**: Асинхронные syscall (epoll/kqueue) — горутины ждут без блока треда.
    
- **Прерывание**: Кооперативное (Go<1.14) — Sysmon ставит флаг stackPreempt в прологах функций/syscall; горутина уступает.

**Конкурентность** — это способность программы **выполнять несколько задач одновременно**, но не обязательно в один и тот же момент времени. Задачи могут переключаться между собой, уступая место друг другу, чтобы эффективно использовать ресурсы.

**Параллелизм** — это **фактическое одновременное выполнение нескольких задач** на разных ядрах или процессорах.
